# Fork Changelog (customizations)

1. **Group-scoped transfer matching** — accounts only auto-match within configured groups; the greedy matcher picks the closest-date pair across all accounts in the same group
2. **Per-family tuning knobs** — configurable date window and FX tolerance (defaults match upstream)
3. **Settings UI** — `/settings/transfer_match_groups` to manage groups and tuning
4. **Transfer detail panel** — shows each transaction's name and a "View transaction" link for both sides of a transfer
5. **Transfer match filter** — transactions page filter to show only auto-matched (pending) or confirmed transfer transactions
6. **Self-hosted build pipeline** — arm64 image to GHCR, auto-deploys via `sure-deploy`
7. ~~**Rule enrichment ignores locks**~~ — **reverted 2026-06-09**, now matches upstream: rules respect attribute locks. A manually-edited (locked) field — e.g. a category set from the UI — is no longer overwritten by rules on any path (sync, scheduled, or "apply all"). `enrich_attributes`' instance guard skips locked attributes regardless of `source` again, and `family/syncer.rb` applies rules without `ignore_attribute_locks`. The `ignore_attribute_locks` flag now only removes the executor's scope pre-filter; the instance guard still protects locked values.
8. **API: `kind` filter and update** — `GET /api/v1/transactions?kind=…` and `PATCH`/`POST /api/v1/transactions` with `kind` now supported; used by the provider-sync scripts to set native kinds at creation (`Contribution → investment_contribution`, `Withdrawal → funds_movement`). We follow stock Sure P&L semantics — there is no `kind`-reclassification housekeeping.
9. **Batch account sync after rule transfer creation** — `SetAsTransferOrPayment` now collects affected accounts and calls `sync_later` once per unique account after all transfers are built, instead of once per transaction, preventing N×2 sync explosions when rules apply to many transactions
10. **AI auto-categorization uses only subcategories** — `Family::AutoCategorizer` filters to subcategories only (`select(&:subcategory?)`), preventing assignment to parent/group categories
11. **Rule action: "Mark as transfer (no counterpart)"** — new `SetAsFundsMovement` action executor sets `kind: funds_movement` on matched transactions without creating a synthetic counterpart entry. Useful for single-leg transfers to untracked external accounts (off-ramps, external wallets).
12. ~~**"Investments Interest Returns" dashboard widget**~~ — **removed 2026-07-02**. Previously showed annualized interest yield (ARR) for interest-bearing investment accounts. Deleted `InvestmentInterestReturnsStatement`, its partial, and its locale file; removed the ivar and `build_dashboard_sections` entry from `pages_controller.rb`. No longer relevant once the vault/fund accounts moved to synthetic NAV holdings (earnings now show as portfolio return, not `Interest` transactions).
14. **"Investments Full" dashboard widget** — unrealized gains/losses for crypto holdings. Groups all current holdings across visible Investment+Crypto accounts by security, showing per-asset cost basis (converted to family currency at avg cost), current market value, unrealized P&L (amount + %), and portfolio total P&L. Rows sorted by current value. Handles missing cost basis gracefully (shows "-" and a note). Also rolls up uninvested cash across the same accounts into a synthetic "Cash" row (cost basis == current value, so it never affects P&L $ amount), so the widget's total matches the native "Investments" widget's account-balance total (holdings + cash) instead of holdings-only. The total P&L **percent** is computed from holdings only (before the Cash row is added) to match the native widget's `unrealized_gains_trend`, which also excludes cash — otherwise cash's cost basis would dilute the percentage even though cash has zero gain. Only shown when at least one holding or nonzero cash balance exists. Modular/merge-safe: new `InvestmentsFullStatement` PORO + new `pages/dashboard/_investments_full` partial + dedicated locale file `config/locales/views/pages/investments_full/en.yml`; upstream touch is two small additions in `pages_controller.rb` (ivar + one `build_dashboard_sections` entry).
13. **"Currency breakdown" dashboard widget** — net worth distribution by account currency. Groups all visible accounts (included-in-finances for the current user) by their native currency, showing per-currency account count, native net total (assets minus liabilities), the same total converted to the family base currency at today's rates, and each currency's share of net worth (abs-weighted). Vehicle accounts are excluded. Only shown when more than one currency is present. Same modular/merge-safe shape as item 12: new `CurrencyBreakdownStatement` PORO + new `pages/dashboard/_currency_breakdown` partial + dedicated locale file `config/locales/views/pages/currency_breakdown/en.yml`; upstream touch is two small additions in `pages_controller.rb` (ivar + one `build_dashboard_sections` entry).
